# FileServer 功能增强设计文档

**日期：** 2026-07-02
**分支：** `feature/auto-refresh-chat-reconnect`
**状态：** 已确认

---

## 概述

为 FileServer（局域网浏览器文件共享服务）新增三个功能：
1. 主机端文件自动刷新（含详细变更日志）
2. 聊天消息板（服务内文字交流 + 快捷复制）
3. WebSocket 连接状态指示器 + 自动重连 + 手动恢复

---

## 功能一：文件自动刷新

### 范围

仅主机页面（`service.html` / `public/js/service.js`）。

### 机制

- 新增一个 **10 秒** 间隔的轮询器（与活动日志轮询间隔一致）。
- 每次轮询通过 File System Access API（`dirHandle.values()`）重新扫描共享目录。
- 对比维度：`name + size + lastModified`。
- 与上次缓存的元数据做全量 diff，检测三类变更：
  - **新增**（名称出现在新列表中但不在缓存中）
  - **删除**（名称在缓存中但不在新列表中）
  - **修改**（名称相同但 size 或 lastModified 不同）
- 有差异时：
  1. 通过 WebSocket 向所有加入者广播新的 `file-list`
  2. 通过 `POST /api/logs/:serviceId` 记录详细变更日志，格式：
     - "文件列表已更新：新增: a.pdf, b.txt; 删除: draft.txt; 修改: notes.md"
  3. 页面活动日志区域实时显示该条目
- 手动刷新按钮（文件列表区域现有刷新图标）触发一次立即扫描 + 对比。

### 边界情况

- 首次扫描没有缓存 → 不产生日志（仅建立基线）
- 轮询出错（如目录权限丢失）→ 在日志中记录警告，不中断后续轮询
- 服务关闭后停止轮询

## 功能二：聊天消息板

### 范围

主机页面（`service.html`）和加入者页面（`join.html`）。

### 机制

- 新增 WebSocket 消息类型 `chat-message`。
- 消息格式：
  ```json
  {
    "type": "chat-message",
    "text": "消息内容",
    "from": "用户名",
    "time": "HH:MM:SS"
  }
  ```
- **不持久化**到数据库。消息仅在当前服务会话内通过 WebSocket 广播。
- 服务端（`ws-handler.js`）收到 `chat-message` 后广播给同一服务内**所有**客户端（包括发送者自己，发送者据此确认消息已发出）。

### UI 布局

**主机页（service.html）：** 右侧 sidebar 在已有"在线用户"和"活动日志"卡片下方新增"聊天"卡片。三卡片顺序：在线用户 → 活动日志 → 聊天。
- 聊天卡片：消息列表（`max-h-48 overflow-y-auto`）+ 底部输入框 + 发送按钮（Enter 键发送），自动滚动到底部。

**加入者页（join.html）：** 在"共享文件"卡片下方新增"聊天"卡片。布局为：已连接信息栏 → 共享文件 → 聊天。
- 聊天卡片结构与主机页一致。
- 仅在 `connected` 状态下渲染。

每条消息格式：
```
[HH:MM:SS] 用户名: 消息内容  📋
```
- 📋 复制按钮：点击复制该条消息文本到剪贴板。
- 复制成功 → 按钮短暂变为 ✓（1.5 秒后恢复）。

### 边界情况

- 新加入者看不到加入前的历史消息（无持久化）
- 空消息不允许发送（前端校验）
- 消息过长（>2000 字符）截断提示
- 服务关闭后聊天自然终止

## 功能三：连接状态 + 恢复

### 范围

主机页面（`service.html`）和加入者页面（`join.html`）。

### 状态指示器

- **位置：** 替换现有 header 中的状态 badge：
  - `service.html` 的 `#header-badge` → 替换为三状态指示器
  - `join.html` 的 `#header-status` → 替换为三状态指示器
- **显示：** 服务页面在 `active` 状态下显示，加入页面在 `connected` 状态下显示。其他状态使用原有 badge。

| 图标 | 状态 | 含义 |
|------|------|------|
| 🟢 | 已连接 | WebSocket 正常，心跳正常 |
| 🟡 | 连接中 | 正在尝试重连（自动重试中） |
| 🔴 | 已断开 | 重试耗尽，需手动恢复 |

### 心跳机制

- 客户端每 **15 秒** 发送 `{ type: "ping" }`。
- 服务端（`ws-handler.js`）收到 `ping` 后回复 `{ type: "pong" }`（echo）。
- 客户端 **45 秒**（3 次心跳周期）未收到 `pong` → 判定连接已断开。

### 自动重连

- 断开后自动重试 **3 次**，指数退避：
  - 第 1 次：2 秒后
  - 第 2 次：4 秒后
  - 第 3 次：8 秒后
- 重试期间状态为 🟡。
- 3 次全部失败后状态变为 🔴，同时显示"恢复"按钮。

### 恢复按钮

- 仅在 🔴 状态下显示。
- 点击后尝试重新建立 WebSocket 连接：
  - **主机**：用保存的令牌/代码重新发送 `register` 消息 → 成功后重新广播 `file-list`。
  - **加入者**：用保存的代码/用户名重新发送 `join` 消息。
- 成功 → 🟢，失败 → 保持 🔴 并弹出错误提示。

### 边界情况

- 服务端主动关闭（主机退出）→ 加入者显示 🔴 但不自动重试（收到 `host-left` 后 redirect 到首页）。
- 用户主动离开（点击"离开"）→ 不触发重连。
- 网络恢复后重连成功，如果服务已过期/关闭 → 显示错误并引导回首页。

## 服务端变更

### `src/ws-handler.js`

新增消息处理：
- `ping` → 回复 `pong`
- `chat-message` → 广播给同一服务内所有客户端（包括发送者）

### 无需新增路由或数据库表

- 聊天消息不持久化
- 文件变更日志复用现有 `POST /api/logs/:serviceId` 路由
- 连接状态纯客户端逻辑

---

## 客户端变更总览

| 文件 | 变更内容 |
|------|----------|
| `public/js/app.js` | 提取共享逻辑：WebSocket 连接管理（心跳、重连、状态管理）、连接状态 UI 组件 |
| `public/js/service.js` | 新增文件轮询模块、聊天模块集成、移除内联的 WS 连接逻辑改用共享模块 |
| `public/js/join.js` | 新增聊天模块集成、移除内联的 WS 连接逻辑改用共享模块 |
| `public/service.html` | 右侧面板拆分（日志 + 聊天）、右上角状态指示器、恢复按钮 |
| `public/join.html` | 右上角状态指示器、恢复按钮、聊天区（放在合适位置） |
| `src/ws-handler.js` | ping/pong 处理、chat-message 广播 |

---

## 非功能需求

- 所有变更为纯前端增量，无需新增 npm 依赖。
- 向后兼容 — 旧版加入者（如缓存的旧 HTML）不影响核心文件共享功能。
- 聊天和文件共享共用同一 WebSocket 连接，不额外建立连接。
